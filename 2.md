# Prompt: Generate AI-Context MD Files for This Project

You are generating a set of Markdown documents that will serve as persistent context for AI coding assistants (Claude Code, Cursor, Windsurf, Copilot, etc.) working on this repository. The goal is to encode knowledge that isn’t obvious from the code, prevent wrong assumptions, and reduce token churn on every future task.

## Phase 1 — Discovery (do this before writing anything)

Inspect the repo and gather:

- Language(s), framework(s), runtime versions, package manager, build tool
- Top-level folder layout and module boundaries
- Lint, format, type-check, and test commands (read package.json, pyproject.toml, Makefile, justfile, CI configs)
- Existing conventions visible in code: naming, error handling, logging, async patterns, import order
- Domain vocabulary used in identifiers, comments, and docs
- External services, databases, queues, and how they’re invoked
- Existing docs (README, /docs, ADRs, wiki links) — extract, don’t duplicate
- Secrets handling pattern (env vars, vault, .env.example)

If a critical fact is unknown, write it inline as `> ASSUMPTION: <X>` at the top of the relevant file so a human can correct it. Do not invent details.

## Phase 2 — Generate these files

Produce each as a separate file. Keep every file under ~300 lines; split if needed.

### 1. `CLAUDE.md` (also acceptable: `AGENTS.md`)

The single most important file. The AI reads this first on every session.

- One-paragraph project summary: what it does, who uses it
- Stack with exact versions
- Local setup: install, build, test, run — copy-pasteable commands only
- **DO** list: idiomatic patterns to follow in this repo
- **DON’T** list: anti-patterns, deprecated paths, dead code areas
- **NEVER without explicit confirmation:** db migrations, deploys, `force-push`, `rm -rf`, secret rotation, anything touching prod
- Where new code goes by concern (auth → `x/`, db access → `y/`, etc.)
- How to write tests here (framework, location, naming, what to mock vs. what to hit for real)
- When to ask the human vs. proceed: explicit thresholds

### 2. `ARCHITECTURE.md`

- Mermaid or ASCII diagram of major components
- Component responsibilities (one paragraph each)
- Data flow for the 2–3 most important journeys
- Storage layer: schemas, key indexes, cache layers, retention
- External integrations and failure modes
- What is **out of scope** for this repo

### 3. `CONVENTIONS.md`

- File/folder naming rules
- Identifier naming (functions, classes, vars, constants, types)
- Error handling: exceptions vs. result types vs. error codes — pick one and say why
- Logging: levels, structured fields, correlation IDs
- Async/concurrency idioms used here
- Commit message format and PR template expectations

### 4. `GLOSSARY.md`

- Every non-obvious domain term with a 1–2 sentence definition
- All acronyms expanded
- Pairs of similar-sounding terms distinguished side-by-side

### 5. `DECISIONS.md` (lightweight ADR log)

Table or sections, each with: Date · Decision · Context · Alternatives considered · Consequences. Include “why we don’t use X” entries for the obvious alternatives a newcomer (or AI) would reach for.

### 6. `TESTING.md`

- Definitions of unit / integration / e2e **as used here**
- Coverage targets and what’s exempt
- Fixture and factory patterns
- How to run a single test, debug a failure, update snapshots, regenerate fixtures

### 7. `PROJECT_STRUCTURE.md`

- Directory tree, top 2–3 levels, one line per folder
- Quick lookup section: “Authentication lives in… Migrations live in… Public API surface lives in…”

## Style rules for every generated file

- Concrete over abstract. Use **real** file paths, **real** command strings, **real** type and function names from this codebase.
- No filler. Strip anything an experienced engineer wouldn’t need.
- Code blocks for every command and code snippet.
- Header at the top of each file: generated date, branch/commit, and a one-line “purpose of this file.”
- Mark anything uncertain with `<!-- REVIEW: ... -->` so a human can scan for it.
- Cross-link files with relative paths instead of repeating content.
- No marketing tone. Write for someone who needs to ship a fix in 20 minutes.

## Output

Emit each file in its own fenced block labeled with the filename, in the order listed above. After the last file, print a short checklist of `<!-- REVIEW -->` markers and `> ASSUMPTION` lines so a human can verify them quickly.
