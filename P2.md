# COBOL → Java Conversion Traceability & Coverage Plan

## Project Apollo — Modernization Assurance

**Objective:** Establish fool-proof traceability between legacy COBOL and converted Java code, with quantified coverage metrics, behavioral validation, and a formal decommission gate.

**Infrastructure:** 100% reuse of existing Apollo stack — Postgres/pgvector, Apache AGE, Ollama (nomic-embed-code), Tree-sitter, Node.js orchestrator, Vitest 3. Zero new external dependencies.

**Timeline:** 8 weeks to full coverage dashboard + ongoing governance.

-----

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                    COVERAGE COMPOSITE SCORE                     │
│   0.20×Structural + 0.20×Semantic + 0.25×Graph + 0.35×Behavioral│
│              Decommission Gate: Each layer ≥ 95%                │
└──────────┬──────────┬───────────┬──────────┬────────────────────┘
           │          │           │          │
    ┌──────▼──┐ ┌─────▼───┐ ┌────▼────┐ ┌──▼──────────┐
    │Structural│ │Semantic │ │  Graph  │ │ Behavioral  │
    │ Layer 1  │ │ Layer 2 │ │ Layer 3 │ │  Layer 4    │
    │  (20%)   │ │  (20%)  │ │  (25%)  │ │   (35%)     │
    ├──────────┤ ├─────────┤ ├─────────┤ ├─────────────┤
    │Tree-sitter│ │nomic-   │ │Apache  │ │Parallel I/O │
    │AST parse │ │embed-   │ │AGE call│ │parity test  │
    │+ naming  │ │code +   │ │chain   │ │COBOL vs Java│
    │rules     │ │pgvector │ │isomor- │ │byte-level   │
    │          │ │cosine   │ │phism   │ │diff         │
    └──────────┘ └─────────┘ └────────┘ └─────────────┘
```

Each layer catches what the previous layers miss:

- **Structural** — Direct 1:1 mappings, naming convention matches
- **Semantic** — Refactored logic, renamed constructs, fuzzy matches
- **Graph** — Broken call chains, missing dependencies, data flow gaps
- **Behavioral** — Logic errors, edge case failures, numeric precision drift

## Phase 1: Inventory & Catalog

**Duration:** Week 1–2
**Goal:** Build a complete structural inventory of both COBOL and Java codebases into Postgres tables.

### 1.1 COBOL Source Parsing

Parse every COBOL source file using Tree-sitter (`tree-sitter-cobol` grammar). Extract all structural constructs.

**Constructs to extract:**

- `PROGRAM-ID`
- `SECTION` and `PARAGRAPH` names
- `COPY` statements (copybook references)
- Data items (`WORKING-STORAGE`, `LINKAGE`, `FILE SECTION`)
- `PERFORM` targets
- `CALL` targets (inter-program calls)
- File descriptors (`SELECT`/`ASSIGN`)
- Embedded SQL (`EXEC SQL` blocks)

**Schema:**

```sql
CREATE TABLE cobol_constructs (
    id                    SERIAL PRIMARY KEY,
    file_path             TEXT NOT NULL,
    program_id            TEXT,
    construct_type        TEXT NOT NULL,
        -- 'program', 'section', 'paragraph', 'copybook',
        -- 'data_item', 'file_descriptor', 'sql_block',
        -- 'call_target', 'perform_target'
    construct_name        TEXT NOT NULL,
    start_line            INT,
    end_line              INT,
    loc                   INT,
    parent_construct_id   INT REFERENCES cobol_constructs(id),
    raw_source            TEXT,
    content_hash          TEXT,      -- MD5 for deduplication
    metadata              JSONB,     -- verb counts, complexity metrics
    created_at            TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_cobol_type ON cobol_constructs(construct_type);
CREATE INDEX idx_cobol_program ON cobol_constructs(program_id);
CREATE INDEX idx_cobol_hash ON cobol_constructs(content_hash);
```

**Implementation notes:**

- Use `@tree-sitter/cobol` via the Node.js orchestrator
- Walk the AST recursively, tracking parent-child relationships
- Compute `content_hash` as MD5 of `raw_source` for dedup detection
- Store verb distribution (COMPUTE, EVALUATE, PERFORM, READ, WRITE, CALL) in `metadata` JSONB

### 1.2 Java Target Parsing

Parse every generated Java file using Tree-sitter (`tree-sitter-java`). Extract all structural constructs.

**Constructs to extract:**

- Packages and class names
- Methods (with full signatures)
- Fields and constants
- Import statements
- Annotations
- Exception handlers (try/catch)
- SQL/JDBC calls
- Inner classes

**Schema:**

```sql
CREATE TABLE java_constructs (
    id                    SERIAL PRIMARY KEY,
    file_path             TEXT NOT NULL,
    package_name          TEXT,
    class_name            TEXT,
    construct_type        TEXT NOT NULL,
        -- 'class', 'method', 'field', 'import',
        -- 'annotation', 'exception_handler',
        -- 'jdbc_call', 'inner_class'
    construct_name        TEXT NOT NULL,
    signature             TEXT,      -- full method signature
    start_line            INT,
    end_line              INT,
    loc                   INT,
    parent_construct_id   INT REFERENCES java_constructs(id),
    raw_source            TEXT,
    content_hash          TEXT,
    metadata              JSONB,
    created_at            TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_java_type ON java_constructs(construct_type);
CREATE INDEX idx_java_class ON java_constructs(class_name);
CREATE INDEX idx_java_hash ON java_constructs(content_hash);
```

### 1.3 Baseline Metrics Snapshot

Compute aggregate metrics and store as a versioned snapshot for trend tracking.

**Metrics to capture:**

|Side  |Metrics                                                                  |
|------|-------------------------------------------------------------------------|
|COBOL |Total programs, sections, paragraphs, copybooks, LOC, verb distribution  |
|Java  |Total classes, methods, fields, LOC, annotation distribution             |
|Ratios|COBOL programs : Java classes, COBOL paragraphs : Java methods, LOC ratio|

**Schema:**

```sql

```

- [ ] All COBOL source files parsed, zero parse failures

- [ ] Baseline snapshot stored and verified

## Phase 2: Deterministic Mapping

**Duration:** Week 2–3
**Goal:** Establish hard links between COBOL constructs and Java constructs using naming conventions, structural patterns, and Windsurf conversion metadata.

### 2.1 Traceability Map Table

Central table linking COBOL to Java constructs with confidence levels.

```sql
CREATE TABLE traceability_map (
    id                  SERIAL PRIMARY KEY,
    cobol_construct_id  INT REFERENCES cobol_constructs(id),
    java_construct_id   INT REFERENCES java_constructs(id),
    mapping_type        TEXT NOT NULL,
        -- 'name_match', 'structure_match', 'semantic_match',
        -- 'manual', 'windsurf_metadata'
    confidence          TEXT NOT NULL,
        -- 'high', 'medium', 'low', 'unmatched'
    similarity_score    NUMERIC,
    notes               TEXT,
    verified_by         TEXT,           -- manual reviewer
    verified_at         TIMESTAMPTZ,
    created_at          TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_trace_cobol ON traceability_map(cobol_construct_id);
CREATE INDEX idx_trace_java ON traceability_map(java_construct_id);
CREATE INDEX idx_trace_confidence ON traceability_map(confidence);
```

### 2.2 Convention-Based Matching

Windsurf follows naming conventions during conversion. Build matchers for common patterns.

**Matching rules (priority order):**

|COBOL Construct        |Expected Java Target    |Normalization                          |
|-----------------------|------------------------|---------------------------------------|
|`PROGRAM-ID`           |Java class name         |`CALC-INTEREST` → `CalcInterest`       |
|`SECTION` / `PARAGRAPH`|Method name             |`100-INIT` → `init()` or `initialize()`|
|`COPYBOOK`             |DTO / Record class      |`CUST-REC.cpy` → `CustRec.java`        |
|`FILE SECTION`         |I/O handler class       |`CUSTOMER-FILE` → `CustomerFileHandler`|
|`WORKING-STORAGE` item |Class field             |`WS-TOTAL-AMT` → `totalAmt`            |
|`CALL` target          |Service/dependency class|`CALL 'VALIDATE'` → `ValidateService`  |

**Normalization algorithm:**

```
1. Strip numeric prefixes (100-, 200-)
2. Replace hyphens with camelCase boundaries
3. Apply case-insensitive comparison
4. Score: exact match = 1.0, prefix match = 0.9, fuzzy (Levenshtein ≤ 2) = 0.8
```

**Output:** Populated `traceability_map` rows with `confidence = 'high'`

### 2.3 Structural Pattern Matching

Beyond names, match by structural signatures derived from AST node-type sequences.

**Pattern fingerprints:**

- COBOL paragraph with 3 `PERFORM`s + 1 `COMPUTE` → Java method with 3 method invocations + 1 arithmetic expression
- `COPY` → `import` chain alignment
- `CALL` → dependency injection / constructor chain
- File I/O pattern: `OPEN` / `READ` / `WRITE` / `CLOSE` → `BufferedReader` / `InputStream` / `OutputStream` patterns
- `EVALUATE` → `switch` statement with same branch count
- `PERFORM UNTIL` → `while` loop

**Output:** Additional `traceability_map` rows with `confidence = 'medium'`

### 2.4 Windsurf Metadata Extraction

Parse any conversion artifacts left by Windsurf as the highest-confidence source.

**Sources to check:**

- Inline comments: `// Converted from: CALC-INTEREST.COMPUTE-RATE`
- `.windsurf` configuration or log files
- Git commit messages from the conversion session (search for patterns like `convert`, `migrate`, `from COBOL`)
- Any mapping/manifest files in the repo

**Output:** `traceability_map` rows with `mapping_type = 'windsurf_metadata'`, `confidence = 'high'`

**Exit criteria for Phase 2:**

- [ ] All naming-convention matches populated
- [ ] Structural pattern matches populated
- [ ] Windsurf metadata extracted (if available)
- [ ] Coverage gap list generated (unmapped COBOL constructs)

## Phase 3: Semantic Embedding Layer

**Duration:** Week 3–4
**Goal:** Use `nomic-embed-code` to catch what deterministic matching missed — fuzzy matches, refactored logic, and semantic gaps.

### 3.1 Embed COBOL Constructs

For every COBOL paragraph/section with >5 LOC, generate an embedding using `nomic-embed-code` (GGUF Q4_K_M, 3584 dims) via Ollama.

**Embedding strategy:**

- Prefix each chunk with: `search_document: COBOL paragraph <name> in <program>: `
- Chunk boundary = paragraph/section boundary (natural COBOL units)
- Skip trivial paragraphs (≤5 LOC, e.g., `STOP RUN` only)

**Schema:**

```sql
CREATE TABLE cobol_embeddings (
    id              SERIAL PRIMARY KEY,
    construct_id    INT REFERENCES cobol_constructs(id),
    embedding       vector(3584),
    chunk_text      TEXT,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_cobol_emb ON cobol_embeddings
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);
```

### 3.2 Embed Java Constructs

Same process for Java methods/classes with >5 LOC.

**Embedding strategy:**

- Prefix: `search_document: Java method <class>.<method>: `
- Chunk boundary = method body (natural Java unit)
- Include method signature in the chunk for context

**Schema:**

```sql
CREATE TABLE java_embeddings (
    id              SERIAL PRIMARY KEY,
    construct_id    INT REFERENCES java_constructs(id),
    embedding       vector(3584),
    chunk_text      TEXT,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_java_emb ON java_embeddings
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);
```

### 3.3 Cross-Language Nearest Neighbor Search

For every COBOL construct NOT yet mapped with `confidence = 'high'`, find top-3 nearest Java neighbors.

**Threshold logic:**

|Cosine Similarity|Action                |Confidence |
|-----------------|----------------------|-----------|
|> 0.85           |Auto-map              |`medium`   |
|0.70 – 0.85      |Flag for manual review|`low`      |
|< 0.70           |Mark as coverage gap  |`unmatched`|

**Reverse search:** Also find Java constructs with no nearby COBOL neighbor — these are potentially dead code or newly added logic not present in the original.

**Coverage gaps table:**

```sql
CREATE TABLE coverage_gaps (
    id                  SERIAL PRIMARY KEY,
    side                TEXT NOT NULL,
        -- 'cobol_unmapped' or 'java_orphan'
    construct_id        INT NOT NULL,
    nearest_match_id    INT,
    nearest_similarity  NUMERIC,
    gap_type            TEXT,
        -- 'missing_conversion', 'partial_conversion',
        -- 'logic_drift', 'dead_code'
    severity            TEXT,       -- 'critical', 'major', 'minor'
    resolution          TEXT,
    resolved_by         TEXT,
    resolved_at         TIMESTAMPTZ,
    created_at          TIMESTAMPTZ DEFAULT NOW()
);
```

**Severity classification:**

- **Critical:** COBOL paragraph with >50 LOC, contains business logic (COMPUTE, EVALUATE), no Java match
- **Major:** COBOL paragraph with 20–50 LOC, moderate complexity, weak Java match
- **Minor:** COBOL paragraph with <20 LOC, utility/boilerplate, no match but low risk

**Exit criteria for Phase 3:**

- [ ] All non-trivial COBOL constructs embedded
- [ ] All non-trivial Java constructs embedded
- [ ] Cross-language matching complete
- [ ] Coverage gaps table populated with severity ratings
- [ ] Manual review queue generated for 0.70–0.85 similarity items

## Phase 4: Graph-Based Dependency Validation

**Duration:** Week 4–5
**Goal:** Verify that COBOL call chains and data flows are preserved in Java using Apache AGE.

### 4.1 COBOL Call Graph

Create the COBOL dependency graph in Apache AGE.

**Node labels:**

- `CobolProgram` — top-level program (PROGRAM-ID)
- `CobolSection` — SECTION within a program
- `CobolParagraph` — PARAGRAPH within a section
- `CobolCopybook` — COPY member
- `CobolFile` — file descriptor (SELECT/ASSIGN)
- `CobolDataItem` — significant data items (level 01, 77)

**Edge labels:**

- `PERFORMS` — paragraph → paragraph
- `CALLS` — program → program (CALL statement)
- `COPIES` — program → copybook
- `READS` — paragraph → file
- `WRITES` — paragraph → file
- `USES` — paragraph → data item
- `MODIFIES` — paragraph → data item

**Example Cypher:**

```cypher
-- Create program node
CREATE (:CobolProgram {
    name: 'CALC-INTEREST',
    file: 'CALC-INTEREST.cbl',
    construct_id: 42
})

-- Create call edge
MATCH (a:CobolParagraph {name: '100-INIT'}),
      (b:CobolParagraph {name: '200-COMPUTE'})
CREATE (a)-[:PERFORMS]->(b)
```

### 4.2 Java Call Graph

Create the Java dependency graph in the same Apache AGE instance.

**Node labels:**

- `JavaClass`
- `JavaMethod`
- `JavaField`
- `JavaInterface`

**Edge labels:**

- `CALLS` — method → method
- `IMPORTS` — class → class
- `EXTENDS` — class → class
- `IMPLEMENTS` — class → interface
- `USES_FIELD` — method → field
- `THROWS` — method → exception class

### 4.3 Cross-Graph Traceability Edges

Using the `traceability_map`, create `CONVERTS_TO` edges between COBOL and Java nodes.

```cypher
-- Link mapped constructs
MATCH (cp:CobolParagraph {construct_id: 42}),
      (jm:JavaMethod {construct_id: 108})
CREATE (cp)-[:CONVERTS_TO {
    confidence: 'high',
    mapping_type: 'name_match',
    similarity: 0.95
}]->(jm)
```

### 4.4 Call Chain Isomorphism Validation

Verify that every COBOL call chain has a corresponding Java call chain.

**Validation query — find broken chains:**

```cypher
MATCH (cp:CobolParagraph)-[:PERFORMS]->(ct:CobolParagraph)
WHERE NOT EXISTS {
    MATCH (cp)-[:CONVERTS_TO]->(jp:JavaMethod),
          (ct)-[:CONVERTS_TO]->(jt:JavaMethod),
          (jp)-[:CALLS]->(jt)
}
RETURN cp.name AS cobol_caller,
       ct.name AS cobol_callee,
       'BROKEN_CHAIN' AS issue
```

**Additional validations:**

|Check                 |Query Logic                                                                           |
|----------------------|--------------------------------------------------------------------------------------|
|Fan-out preservation  |COBOL paragraph with N PERFORMs → Java method should have ~N calls                    |
|Fan-in preservation   |COBOL paragraph called by N others → Java method called by ~N                         |
|Data flow preservation|If COBOL reads FILE-A in para X, writes FILE-B in para Y → Java equivalents must exist|
|Copybook dependency   |Every COPY → corresponding import chain must exist                                    |
|Inter-program calls   |Every CALL → corresponding class instantiation / injection                            |
|SQL equivalence       |EXEC SQL blocks → JDBC/JPA calls with same table references                           |

**Output:** Graph validation report stored in:

```sql
CREATE TABLE graph_validation_results (
    id              SERIAL PRIMARY KEY,
    validation_type TEXT NOT NULL,
    cobol_source    TEXT,
    java_target     TEXT,
    status          TEXT NOT NULL,     -- 'pass', 'fail', 'warning'
    detail          JSONB,
    run_at          TIMESTAMPTZ DEFAULT NOW()
);
```

**Exit criteria for Phase 4:**

- [ ] COBOL call graph fully populated
- [ ] Java call graph fully populated
- [ ] CONVERTS_TO edges created from traceability_map
- [ ] All validation queries executed
- [ ] Broken chains documented in coverage_gaps

## Phase 6: Coverage Dashboard & Reporting

**Duration:** Week 7–8
**Goal:** Build a real-time dashboard showing conversion coverage, gaps, and verification status across all layers.

### 6.1 Coverage Metrics Engine

Materialized views for each coverage layer.

```sql
CREATE MATERIALIZED VIEW coverage_summary AS

-- Structural coverage
SELECT
    'structural' AS layer,
    COUNT(*) FILTER (WHERE confidence IN ('high', 'medium'))::NUMERIC
        / NULLIF(COUNT(*), 0) * 100 AS coverage_pct,
    COUNT(*) FILTER (WHERE confidence = 'unmatched') AS gap_count
FROM traceability_map

UNION ALL

-- Semantic coverage
SELECT
    'semantic',
    COUNT(*) FILTER (WHERE similarity_score > 0.85)::NUMERIC
        / NULLIF(COUNT(*), 0) * 100,
    COUNT(*) FILTER (WHERE similarity_score < 0.70 OR similarity_score IS NULL)
FROM traceability_map
WHERE mapping_type = 'semantic_match'

UNION ALL

-- Graph coverage
SELECT
    'graph',
    COUNT(*) FILTER (WHERE status = 'pass')::NUMERIC
        / NULLIF(COUNT(*), 0) * 100,
    COUNT(*) FILTER (WHERE status = 'fail')
FROM graph_validation_results

UNION ALL

-- Behavioral coverage
SELECT
    'behavioral',
    COUNT(*) FILTER (WHERE match = true)::NUMERIC
        / NULLIF(COUNT(*), 0) * 100,
    COUNT(*) FILTER (WHERE match = false)
FROM behavioral_results;

-- Refresh on demand
REFRESH MATERIALIZED VIEW coverage_summary;
```

**Per-program detail view:**

```sql
CREATE MATERIALIZED VIEW program_coverage AS
SELECT
    cc.program_id,
    COUNT(DISTINCT cc.id) AS total_cobol_constructs,
    COUNT(DISTINCT tm.id) FILTER (WHERE tm.confidence = 'high') AS high_confidence_maps,
    COUNT(DISTINCT tm.id) FILTER (WHERE tm.confidence = 'medium') AS medium_confidence_maps,
    COUNT(DISTINCT cg.id) FILTER (WHERE cg.severity = 'critical') AS critical_gaps,
    COUNT(DISTINCT br.id) FILTER (WHERE br.match = true) AS tests_passing,
    COUNT(DISTINCT br.id) FILTER (WHERE br.match = false) AS tests_failing
FROM cobol_constructs cc
LEFT JOIN traceability_map tm ON tm.cobol_construct_id = cc.id
LEFT JOIN coverage_gaps cg ON cg.construct_id = cc.id AND cg.side = 'cobol_unmapped'
LEFT JOIN behavioral_results br ON br.cobol_program = cc.program_id
GROUP BY cc.program_id;
```

## Phase 7: Continuous Governance

**Duration:** Ongoing
**Goal:** Ensure traceability stays valid as code evolves post-conversion.

### 7.1 Git Hook Integration

On every commit to the Java codebase:

1. Re-parse changed files → update `java_constructs`
1. Recompute embeddings for modified methods → update `java_embeddings`
1. Re-run graph validation for touched methods
1. Alert if a mapped Java method is deleted or significantly refactored (`content_hash` change)

**Hook trigger logic:**

```bash
# post-commit hook (simplified)
CHANGED_FILES=$(git diff --name-only HEAD~1 HEAD -- '*.java')
for file in $CHANGED_FILES; do
    node /apollo/scripts/reparse-java.js "$file"
    node /apollo/scripts/reembed-java.js "$file"
    node /apollo/scripts/validate-graph.js "$file"
done
```

### 7.2 Drift Detection Job

Weekly scheduled job to catch gradual logic divergence.

**Process:**

1. Identify all Java constructs modified in the past 7 days (via `content_hash` comparison)
1. Re-embed modified constructs
1. Recompute similarity against their mapped COBOL counterparts
1. If similarity drops below 0.70 for any pair → raise a drift alert

**Drift alerts table:**

```sql
CREATE TABLE drift_alerts (
    id                  SERIAL PRIMARY KEY,
    cobol_construct_id  INT REFERENCES cobol_constructs(id),
    java_construct_id   INT REFERENCES java_constructs(id),
    previous_similarity NUMERIC,
    current_similarity  NUMERIC,
    drift_delta         NUMERIC,
    alert_level         TEXT,       -- 'warning' (>0.10 drop), 'critical' (>0.20 drop)
    acknowledged_by     TEXT,
    acknowledged_at     TIMESTAMPTZ,
    created_at          TIMESTAMPTZ DEFAULT NOW()
);
```

### 7.3 Decommission Gate

Formal go/no-go checkpoint before any COBOL program is retired.

**Gate requirements (ALL must pass):**

|Requirement        |Threshold      |Verification                            |
|-------------------|---------------|----------------------------------------|
|Structural mapping |100%           |Every paragraph has a mapped Java method|
|Semantic similarity|≥ 0.85 avg     |Average similarity across all mappings  |
|Graph validation   |0 broken chains|All call chains preserved               |
|Behavioral parity  |100% tests pass|All regression + edge case tests passing|
|Manual sign-off    |Recorded       |Tech lead + business owner verified     |
|Drift alerts       |0 open         |No unacknowledged drift alerts          |

**Decommission record:**

```sql
CREATE TABLE decommission_log (
    id                      SERIAL PRIMARY KEY,

    business_signoff        TEXT,
    business_signoff_at     TIMESTAMPTZ,
    legal_signoff           TEXT,
    legal_signoff_at        TIMESTAMPTZ,
```
